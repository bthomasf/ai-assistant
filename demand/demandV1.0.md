## 1. 项目概述

### 1.1 项目定位

本项目是一名**服务端 Java 开发工程师**的个人 AI 助手知识库，旨在系统化地整理、管理和迭代日常工作中使用的各类 AI 能力，打造覆盖工作全生命周期的 AI 百科全书。

### 1.2 核心价值

项目围绕四个关键动作形成闭环：

| 动作 | 说明 |
|------|------|
| **分类** | 按工具形态、工作阶段对 AI 能力进行体系化归类 |
| **唤醒** | 在具体工作场景中快速定位并调用对应的 AI 工具 |
| **协助** | 借助 AI 工具高效完成开发任务 |
| **沉淀** | 将 AI 协助的真实案例回填到项目中，持续积累经验 |
| **可视化** | 通过前端全景视图直观浏览工作生命周期、工具分类和案例库，一目了然 |

### 1.3 解决的核心问题

- AI 工具散落各处，缺乏统一索引，找工具靠记忆
- 用过的好用法没有记录，下次遇到同类问题又要重新摸索
- 缺少一个以"我的工作流"为中心的 AI 能力视图
- 经验无法累积成体系，无法形成个人竞争力
- 纯文档形式不够直观，缺乏可视化的全景浏览和快速检索手段

## 2. 目标用户画像

### 2.1 角色定义

| 维度 | 描述 |
|------|------|
| 职位 | 服务端 Java 开发工程师 |
| 技术栈 | Java / Spring Boot / MyBatis / MySQL / Redis / Kafka / Docker / K8s 等 |
| 工作内容 | 需求开发、接口设计、编码实现、CR、联调测试、上线运维、故障排查 |
| AI 使用习惯 | 已使用 ChatGPT / Claude / Cursor / Copilot 等工具，但缺乏系统化管理 |

### 2.2 工作场景覆盖

本项目关注的是**日常开发全生命周期**，从拿到需求到代码上线，从值班巡检到故障复盘，每个环节都映射对应的 AI 能力。

### 2.3 核心诉求

1. **效率优先**：快速找到能解决当前问题的 AI 工具
2. **经验复用**：过去成功的 AI 用法能被自己（及团队）反复借鉴
3. **持续进化**：随着新工具、新场景出现，知识库能灵活扩展

## 3. 工作生命周期阶段划分

按照 Java 服务端开发的典型工作流，将 AI 能力映射到以下 8 个阶段：

### 3.1 需求分析与方案设计

**阶段描述**：从产品需求文档到技术方案落地的过程。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 需求理解与拆解 | 用 AI 对话梳理需求边界、识别模糊点 |
| 技术方案设计 | 让 AI 生成方案初稿、对比多种架构选型 |
| 数据库表结构设计 | 根据业务模型让 AI 建议表结构、索引策略 |
| 接口契约定义 | AI 辅助生成 OpenAPI/Protobuf 定义 |
| 工时评估 | 基于任务拆解让 AI 辅助估算复杂度 |

### 3.2 编码开发

**阶段描述**：日常编码实现阶段，占工作时间比例最大。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 业务逻辑编写 | IDE 内 AI 补全、生成代码片段 |
| 模板代码生成 | CRUD、DTO、Mapper 等样板代码自动生成 |
| SQL 编写与优化 | AI 辅助写复杂查询、分析执行计划 |
| 正则/脚本编写 | 复杂正则表达式、Shell 脚本生成 |
| 重构建议 | AI 分析代码结构并提供重构方案 |

### 3.3 代码审查

**阶段描述**：提交 MR/PR 前后的代码质量把关。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 自审代码 | AI 审查 diff，发现潜在 bug 和改进点 |
| 审查他人代码 | AI 辅助理解变更意图，定位风险点 |
| 代码规范检查 | 基于团队规范的 AI 检查 |
| Commit 信息规范 | AI 生成规范化的 commit message |

### 3.4 测试与质量保障

**阶段描述**：单元测试、集成测试和质量度量。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 单元测试编写 | AI 根据业务代码生成测试用例 |
| 测试数据构造 | AI 生成边界值、异常场景的 mock 数据 |
| 接口测试 | AI 辅助编写接口测试脚本 |
| 覆盖率分析 | AI 分析未覆盖路径并建议补充测试 |

### 3.5 部署与运维

**阶段描述**：从打包构建到线上部署与日常运维。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| CI/CD 配置 | AI 辅助编写 Jenkinsfile / GitHub Actions |
| Docker/K8s 配置 | Dockerfile、Helm Chart 的生成与优化 |
| 上线检查清单 | AI 生成发布前检查项 |
| 监控告警配置 | Prometheus 规则、Grafana 面板建议 |

### 3.6 文档与知识管理

**阶段描述**：技术文档编写和团队知识沉淀。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 接口文档 | AI 根据代码生成 API 文档 |
| 设计文档 | AI 辅助组织文档结构和内容 |
| 会议纪要 | AI 总结会议要点和待办 |
| 技术分享 | AI 辅助准备分享材料和 PPT 大纲 |

### 3.7 问题排查与故障处理

**阶段描述**：线上问题定位与修复。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 日志分析 | AI 解读异常堆栈、定位根因 |
| 性能排查 | AI 分析慢查询、线程 dump、GC 日志 |
| 故障复盘 | AI 辅助撰写故障复盘报告 |
| 应急脚本 | AI 快速生成数据修复/回滚脚本 |

### 3.8 学习与技能提升

**阶段描述**：技术成长和新技术学习。

| 典型任务 | AI 可协助的方向 |
|----------|----------------|
| 新技术调研 | AI 对比技术选型、总结优劣 |
| 源码阅读 | AI 辅助解读开源框架核心流程 |
| 面试准备 | AI 模拟技术面试、解析高频题 |
| 技术写作 | AI 辅助撰写博客/技术文章 |

## 4. AI工具分类体系

### 4.1 分类维度

按**工具形态**进行一级分类，再按**适用阶段**做二级映射。

### 4.2 AI 平台类

> 通过网页或 API 使用的通用 AI 对话平台。

| 工具名称 | 简介 | 适用阶段 | 备注 |
|----------|------|----------|------|
| ChatGPT | OpenAI 旗舰对话模型 | 全阶段通用 | GPT-4o / o1 等 |
| Claude | Anthropic 出品，长上下文能力强 | 需求分析、文档、方案设计 | Claude 3.5 Sonnet / Opus |
| DeepSeek | 国产高性价比模型 | 编码、翻译、问答 | 开源可私有部署 |
| Gemini | Google 多模态模型 | 图文理解、代码生成 | - |
| 文心一言 / 通义千问 | 国内大厂模型 | 中文场景、合规场景 | - |

### 4.3 IDE 集成类

> 嵌入开发环境中的 AI 编程助手。

| 工具名称 | 简介 | 适用阶段 | 备注 |
|----------|------|----------|------|
| Cursor | AI-first 编辑器，内置多模型 | 编码、审查、重构 | 支持 Agent 模式 |
| GitHub Copilot | GitHub 官方 AI 补全 | 编码、测试 | VS Code / JetBrains |
| JetBrains AI | IDEA 内置 AI 助手 | 编码、重构 | 与 IDE 深度集成 |
| Tabnine | 支持私有部署的补全工具 | 编码 | 适合安全要求高的团队 |
| Cline / Aider | 终端 AI 编程代理 | 编码、重构 | CLI 模式 |

### 4.4 MCP 服务类

> Model Context Protocol 服务，为 AI 模型提供外部工具能力。

| MCP 服务 | 能力 | 适用阶段 | 备注 |
|----------|------|----------|------|
| GitLab MCP | 搜索代码/仓库、管理 MR | 编码、审查 | 内网代码检索 |
| Sourcegraph MCP | 全局代码搜索 | 编码、排查 | 跨仓库搜索 |
| Overmind MCP | 任务/需求管理 | 需求分析、项目管理 | 创建/更新任务 |
| Mlog MCP | 日志检索 | 运维、排查 | 线上日志搜索 |
| OX MCP | API/RPC 检索 | 编码、联调 | 接口文档查询 |
| 知识库 MCP | 内部文档搜索 | 全阶段 | 内网知识检索 |

### 4.5 Skills 能力类

> 预定义的 AI 技能模板，在特定场景下自动触发执行。

| Skill 名称 | 能力 | 适用阶段 | 备注 |
|------------|------|----------|------|
| commit-skill | 分析变更、生成 commit、记录日志 | 编码 | 自动化提交流程 |
| code-review-skill | 代码审查、检测问题、生成报告 | 审查 | 支持 git diff 审查 |
| bug-trace-skill | 根据 bug 现象追溯变更根因 | 排查 | 结合 COMMIT-LOG 分析 |
| daily-report-skill | 扫描 git 提交生成工作日报 | 文档 | 跨项目汇总 |
| commit-review | 变更分组、拆分 commit 建议 | 编码 | 规范化提交 |
| diagram 系列 | Mermaid/PlantUML/SVG 等图表生成 | 设计、文档 | 多种输出格式 |
| doc-coauthoring | 文档协同撰写 | 文档 | 结构化写作流程 |

### 4.6 自动化脚本 / Agent 类

> 具备自主执行能力的 AI 代理或脚本工具。

| 工具名称 | 能力 | 适用阶段 | 备注 |
|----------|------|----------|------|
| Cursor Agent 模式 | 多步骤自主编码 | 编码、重构 | 可调用终端和工具 |
| Claude Code | 终端 AI 代理 | 编码、运维 | CLI 环境 |
| 自定义 Shell 脚本 | 批量操作自动化 | 运维、部署 | AI 辅助生成 |
| GitHub Actions AI | CI/CD 流水线集成 | 部署 | 自动化工作流 |

## 5. 案例沉淀机制

### 5.1 设计目标

每一次有价值的 AI 协助都应被记录下来，形成可复用的经验资产。

### 5.2 案例模板

每个案例统一采用以下 Markdown 模板：

```markdown
# [案例标题]

## 基本信息

| 字段 | 值 |
|------|-----|
| 日期 | YYYY-MM-DD |
| 工作阶段 | （如：编码开发 / 故障排查） |
| 使用工具 | （如：Cursor Agent + GitLab MCP） |
| 难度评级 | ⭐ ~ ⭐⭐⭐⭐⭐ |
| 耗时 | （如：30 分钟） |

## 场景描述

> 遇到了什么问题 / 要完成什么任务？

## AI 协助过程

> 使用了哪些 AI 工具？怎么用的？关键 Prompt 是什么？

## 最终结果

> 产出了什么？效果如何？

## 经验总结

> 学到了什么？下次可以怎么优化？

## 关联标签

`#阶段标签` `#工具标签` `#技术标签`
```

### 5.3 案例回填流程

```
完成 AI 协助任务
      │
      ▼
  值得记录？──否──▶ 跳过
      │
     是
      ▼
  从模板创建案例文件
      │
      ▼
  填写核心内容
  （可用 AI 辅助撰写）
      │
      ▼
  添加标签和分类
      │
      ▼
  存入 cases/ 目录
      │
      ▼
  定期回顾，提炼通用模式
```

### 5.4 案例命名规范

文件命名格式：`YYYY-MM-DD_[阶段简写]_[简要描述].md`

示例：
- `2026-03-09_coding_cursor-agent-batch-refactor.md`
- `2026-03-08_debug_oom-heap-dump-analysis.md`
- `2026-03-07_design_redis-cache-strategy.md`

阶段简写对照：

| 简写 | 阶段 |
|------|------|
| `design` | 需求分析与方案设计 |
| `coding` | 编码开发 |
| `review` | 代码审查 |
| `test` | 测试与质量保障 |
| `deploy` | 部署与运维 |
| `doc` | 文档与知识管理 |
| `debug` | 问题排查与故障处理 |
| `learn` | 学习与技能提升 |

### 6.1 目录设计原则

| 原则 | 说明 |
|------|------|
| **按关注点分离** | 工具目录 (catalog)、生命周期指南 (lifecycle)、实战案例 (cases) 三大核心区域互不耦合 |
| **按阶段可索引** | 通过 lifecycle/ 目录可以快速找到某阶段适用的全部 AI 工具 |
| **按工具可查阅** | 通过 catalog/ 目录可以深入了解单个 AI 工具的用法和配置 |
| **案例可追溯** | cases/ 按年月归档，命名含日期和阶段标签，便于检索 |
| **模板驱动** | 案例、工具卡片均有统一模板，降低记录门槛 |
| **前端独立** | `web/` 为独立前端工程，拥有自己的 package.json 和构建流程，不侵入项目其他目录 |
| **数据生成** | 前端数据由构建脚本从 MD 文件自动生成，保证内容与文档单一数据源 |

## 7. 可视化前端工程

### 7.1 功能概述

项目内置一个基于 React + Vite + TailwindCSS 的单页应用，核心目标是以可视化方式浏览工作生命周期、AI 工具分类、案例库、Prompt 库等全部项目内容。

| 特征 | 说明 |
|------|------|
| 架构形式 | 纯前端静态站点，无需后端服务 |
| 构建方式 | `npm run build` 后可直接在浏览器打开 |
| 数据来源 | 项目 Markdown 文件，构建时自动解析为 JSON |

### 7.2 技术选型

| 技术 | 用途 |
|------|------|
| React 18 + TypeScript | 前端框架与类型安全 |
| Vite | 构建工具，开发热更新 |
| TailwindCSS | 样式方案，实用优先的 CSS 框架 |
| React Router | 客户端路由 |
| remark / gray-matter | 解析 Markdown + Frontmatter |
| Node.js 构建脚本 | build 前扫描 `catalog/`、`lifecycle/`、`cases/`、`prompts/` 目录，解析 MD 文件生成 `src/data/*.json` |

### 7.3 页面规划

共 5 个核心页面：

#### 7.3.1 Dashboard 首页

- 工作生命周期全景图（8 阶段环形/流程图）
- 各阶段工具数量统计
- 最近案例概览

#### 7.3.2 生命周期视图

- 8 个阶段卡片式展示
- 点击进入详情查看典型任务、推荐工具、关联案例

#### 7.3.3 AI 工具目录

- 按 5 大分类（平台 / IDE / MCP / Skills / Agent）展示工具卡片列表
- 支持分类筛选和关键词搜索

#### 7.3.4 案例浏览

- 按时间线 / 阶段 / 工具多维度浏览已沉淀案例
- 直接渲染 Markdown 内容

#### 7.3.5 Prompt 集锦

- 按阶段分类展示高质量 Prompt
- 一键复制 Prompt 内容

### 7.4 数据流设计

```
catalog/*.md  ─┐
lifecycle/*.md ─┤  Node 构建脚本    src/data/*.json    React 组件渲染
cases/*.md     ─┤  ─────────────▶  ──────────────▶   页面展示
prompts/*.md   ─┘  (prebuild)       (静态 import)
```

**数据流说明**：

1. **构建阶段**：Node.js 脚本在 `npm run build` 前执行，扫描项目中的 Markdown 文件
2. **解析阶段**：通过 gray-matter 提取 Frontmatter 元数据，通过 remark 解析正文内容
3. **输出阶段**：生成结构化的 JSON 文件到 `src/data/` 目录
4. **渲染阶段**：React 组件静态 import JSON 数据并渲染页面

### 7.5 交互设计要求

| 要求 | 说明 |
|------|------|
| 响应式布局 | 支持桌面端和平板，最小适配宽度 768px |
| 主题切换 | 暗色/亮色主题切换，跟随系统偏好或手动切换 |
| 全局搜索 | 跨工具、案例、Prompt 的关键词检索，支持模糊匹配 |
| 阶段导航 | 生命周期阶段之间可导航跳转，点击阶段可查看关联工具和案例 |
| 流畅体验 | 页面切换无刷新，列表支持虚拟滚动（如数据量较大） |
